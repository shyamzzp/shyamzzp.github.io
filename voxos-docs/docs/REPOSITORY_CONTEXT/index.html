VOXos Repository Context

This repository is now focused on the native iOS/Xcode application under voice_xcode/VOXOS.xcodeproj. Expo and older generated app artifacts are historical only; new product work should target the Xcode app.

Current Product Direction​

VOXos is a compact, full-screen iOS dashboard for daily personal context. It combines local device information, file indexing, notes, persona facts, health reports, calls/status triggers, notifications, settings, voice commands, local model configuration, Cerebras/OpenRouter cloud inference, Composio connector setup, and app-level sound/haptic controls. The app also includes a WidgetKit extension for Lock Screen/Home widgets, selected VOXos app shortcut widgets, and ActivityKit Live Activities, backed by a compact App Group snapshot.

The main interaction model is:

  • The dashboard shows compact app icons.
  • Opening an app routes to a sliding detail page.
  • Pinning keeps an app in the persistent list, but newly opened items still appear first.
  • Voice is activated through a long press on the Voice control.
  • Voice commands can route directly to app detail pages, for example open focus, open security, or open calls.
  • Bundled App Shortcuts expose selected VOXos routes to Siri, Spotlight, Shortcuts, and Action Button configuration after install. The shortcut layer lives in VoiceModeShortcut.swift and hands off through AppLaunchRoute.featureDefaultsKey or OpenVoiceModeOnLaunch.
  • Voice commands can request Google/Gmail access; VOXOS checks the active local Google connection and opens Account Integrations/sign-in when access is missing.
  • Voice commands can request Composio/connectors/toolkits access; VOXOS opens the Composio detail page and prompts for API configuration if needed.
  • Voice command behavior can be customized from Settings through an editable AGENTS.md file stored at Application Support/VOXOS/VoiceCommands/AGENTS.md, mirrored in SQLite, and included in local LLM/Cerebras prompts.
  • Debug development installs keep heavy models out of the app bundle and reuse model files seeded once into Application Support/VOXOS/Models.
  • Persona voice and typed entries stage proposed attribute changes first. The user sees confidence/change probability and must approve with OK update before SQLite facts are changed.
  • Local model downloads report live progress in Settings.

Native iOS Stack​

  • UI: SwiftUI.
  • Local persistence: SQLite via LocalStore.
  • Local GGUF inference: bundled voice_xcode/Frameworks/llama.xcframework.
  • Bundled local LLM: voice_xcode/VOXOS/BundledLocalModels/SmolLM2-135M-Instruct-Q4_K_M.gguf.
  • Local speech-to-text: bundled Parakeet TDT 0.6B v3 Core ML through FluidAudio.
  • Cloud inference: backend mode, direct Cerebras configuration, and OpenRouter chat completions.
  • Google OAuth: GoogleOAuthClient with ASWebAuthenticationSession, local SQLite config/status, multiple connected Google accounts, an active account pointer, and Gmail read-only scope foundation.
  • Google Calendar: GoogleCalendarClient fetches upcoming primary-calendar events for every native Google account with calendar.readonly; Schedule also aggregates connected Composio Google Calendar accounts for the configured VOXos user. Refreshes replace the SQLite schedule cache, including empty/failure refreshes, so stale events from a previously selected account do not remain. Schedule stores normalized event fields plus provider/source provenance such as Google event source links, Gmail/fromGmail indicators when returned, and compact Composio payload details. The UI renders a compact calendar strip plus date-grouped rows, opens a scrollable source-aware details sheet, and sanitizes provider failures into design-system alerts rather than showing raw JSON. Schedule event intelligence classifies common event types, extracts practical fields from description/source payloads, and optionally uses the configured local/cloud LLM for a compact meaning summary.
  • Composio connector setup: local SQLite config for API key, VOXos user ID, default toolkit, and base URL, plus a Composio connected-account test/list flow.
  • Health reports: HealthKit plus Swift Charts.
  • Widgets/Live Activities: WidgetKit and ActivityKit through VOXOSWidgetExtension, with shared snapshot data in App Group group.com.shyamzzp.voxos. VOXos Status shows summary state; VOXos Apps shows user-selected app shortcuts that deep-link into matching VOXos detail pages.
  • System shortcuts: App Intents/App Shortcuts through VoiceModeShortcut.swift.
  • Device install helper: scripts/install_wifi_iphone.sh.
  • Development model cache helper: scripts/seed_dev_models_to_iphone.sh.
  • UI smoke helper: scripts/ui_smoke_dashboard_navigation.sh.

Bundled Parakeet v3 Voice Input​

Parakeet v3 is used as the default packaged voice input engine. It is an ASR/STT model, not a reasoning LLM. The app still uses rules, imported/downloaded local LLMs, or cloud providers for response generation.

Bundled source:

  • voice_xcode/VOXOS/BundledVoiceModels/parakeet-tdt-0.6b-v3-coreml/

Runtime loading:

  • VOXos first checks Application Support/VOXOS/Models/Voice/parakeet-tdt-0.6b-v3-coreml, then falls back to the packaged .mlmodelc files in the app bundle.
  • VOXos constructs FluidAudio's AsrModels object manually after validating the bundled file set and weight sizes.
  • VOXos prewarms Parakeet while the user is recording, shows progress text during transcription, and falls back to Apple file transcription if Parakeet does not return quickly.

Why VOXos bypasses FluidAudio's downloader/cache loader:

  • The app bundle is read-only on iOS.
  • FluidAudio's loader can attempt model-cache repair or download if any required file appears missing.
  • Letting that loader point at bundled resources can therefore fail with a permission error.
  • Direct Core ML loading avoids write attempts.
  • No user Files permission is required for bundled Parakeet voice input.

Required bundled files:

  • Preprocessor.mlmodelc
  • Encoder.mlmodelc
  • Decoder.mlmodelc
  • JointDecisionv3.mlmodelc
  • parakeet_vocab.json

Bundled SmolLM2 Local LLM​

SmolLM2 135M Instruct Q4_K_M is packaged as the default GGUF local LLM for short offline replies.

Bundled source:

  • voice_xcode/VOXOS/BundledLocalModels/SmolLM2-135M-Instruct-Q4_K_M.gguf

Runtime loading:

  • Debug builds exclude BundledLocalModels; Release builds can still copy it into the app bundle.
  • VOXos first checks Application Support/VOXOS/Models/Local/SmolLM2-135M-Instruct-Q4_K_M.gguf, then falls back to the app bundle.
  • AppModel.configureBundledLocalModel() registers the cached or bundled file in SQLite on launch.
  • If the saved local model is missing, invalid, or None, VOXOS selects the bundled model, switches voice replies to Local LLM, and persists that choice.
  • If the user has already selected another valid local model, VOXOS preserves that selection.

Storage note:

  • The bundled model remains read-only in the app bundle. Imported, downloaded, and development-seeded models live under Application Support/VOXOS/Models/.

Important Paths​

  • Xcode project: voice_xcode/VOXOS.xcodeproj
  • Main app model: voice_xcode/VOXOS/AppModel.swift
  • Main dashboard UI: voice_xcode/VOXOS/DashboardView.swift
  • Widget extension: voice_xcode/VOXOSWidgetExtension/
  • Widget shared model: voice_xcode/VOXOS/VOXOSWidgetShared.swift
  • Live Activity controller: voice_xcode/VOXOS/VOXOSLiveActivityController.swift
  • Voice controller: voice_xcode/VOXOS/SpeechController.swift
  • SQLite store: voice_xcode/VOXOS/LocalStore.swift
  • Voice mode screen: voice_xcode/VOXOS/VoiceModeView.swift
  • App Shortcuts/App Intents: voice_xcode/VOXOS/VoiceModeShortcut.swift
  • Google OAuth client: voice_xcode/VOXOS/GoogleOAuthClient.swift
  • Voice command instructions: created on device at Application Support/VOXOS/VoiceCommands/AGENTS.md
  • Docs: docs/

Repository Rules For Future Work​

  • Keep changes native/Xcode-first.
  • Keep large model artifacts in Git LFS.
  • Keep feature documentation in docs/ current when behavior changes.
  • Build and install with the Xcode project, not Expo.
  • Do not write runtime data into the app bundle; use Application Support, Caches, or SQLite.
