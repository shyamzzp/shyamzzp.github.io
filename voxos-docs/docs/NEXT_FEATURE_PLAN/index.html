VOXos Next Feature Plan

This plan captures the next product work after the terminal/Mac Link pass. Each item should be implemented in small slices with regular commits, regular PRs, native Xcode verification, squash merge, and local main sync before moving to the next slice.

PR And Commit Cadence​

  • Create one feature branch per small implementation slice, not one large branch for a whole epic.
  • Commit after each coherent working change: model/storage, UI, service integration, tests/docs.
  • Open a PR once a slice builds and has a focused verification result.
  • Squash merge each PR after verification, then fast-forward local main.
  • Start the next slice from updated main.
  • Keep PR titles action-oriented, for example Add command center intent history or Add Persona timeline approvals.
  • Do not batch unrelated UI, storage, provider, and docs changes into the same PR unless they are required for one working slice.

0. VOXos Gateway, Nodes, Skills, And Channels​

Goal: borrow the strongest OpenClaw-style runtime ideas without turning VOXos into a clone. VOXos stays the native iPhone/iPad control surface, while a small always-on Gateway on the Mac owns sessions, trusted nodes, skill/tool profiles, background jobs, and future messaging entry points.

Implementation:

  • Add a VOXos Gateway contract with gateway status, active sessions, node capabilities, tool profiles, skills, channels, and job queues.
  • Fetch the Gateway status contract through Mac Link in the iOS app and show live read-only readiness in Mac Link, Service Health, Jobs, Device Fleet, Browser Tasks, Memory Answer, and Automation Gallery.
  • Treat iPhone, iPad, Mac, Watch, WebChat, Telegram, Discord, and future channel adapters as trusted nodes or channels with explicit capability flags.
  • Turn existing app capabilities into inspectable skills: Mac control, Terminal/Pilotty, Agent Browser, GitHub Scout, Hugging Face Hub, Composio, Files/Search, Receipts, Persona, and Daily Briefing.
  • Add tool profiles such as minimal, read-only, personal, developer, and mac-control; destructive or external actions stay approval-gated.
  • Start with read-only status and local metadata in the iOS app, then connect execution through Mac Link/Gateway once the user approves.
  • Keep messaging channels incremental: WebChat/admin first for debugging, then Telegram or Discord as the first external entry point.

Testing:

  • Verify Gateway status appears in Daemons, Device Fleet, Jobs, Service Health, and global search.
  • Verify Mac Link details can refresh Gateway status and show node, skill, channel, profile, and job counts from the live payload.
  • Verify the Mac Link server exposes a Gateway status payload with nodes, skills, channels, profiles, and jobs.
  • Verify no messaging channel or external command runs automatically from this first slice.
  • Verify future executable Gateway tasks have a clear approval, profile, and audit boundary.

1. Unified Command Center​

Goal: make one voice/text command layer route requests across Terminal, Mac Link, Composio, Google, Persona, Files, Calendar, Wallet, Music, Receipts, and Settings.

Implementation:

  • Create a shared command execution surface that every detail page can call.
  • Normalize command intents into action, target app, provider, confidence, risk, and confirmation requirement.
  • Add command history rows with source app, heard text, interpreted intent, result, and timestamp.
  • Route setup failures to the exact detail page or settings subsection needed to fix them.
  • Make setup and ready health rows tappable so GitHub, Composio, Hugging Face, Mac Link, Terminal, widgets, and other system items open the owning detail page or scroll Settings to the exact setup section.
  • Run terminal commands from ~, stream captured tmux output while the command is still running, preserve terminal color semantics in VOXos, and surface the final response with the exit code when the command completes.

Testing:

  • Say or type open terminal, open composio, refresh schedule, search receipts, and update persona.
  • Verify safe commands run directly and risky/data-changing commands require Confirm/Cancel.
  • Verify failed provider setup opens the correct setup page with a VOXos-styled message.

2. Persona Memory Timeline​

Goal: show what VOXos has learned about the user, where it came from, and whether it should be trusted.

Implementation:

  • Add a timeline view for Persona facts with source, confidence, created date, last updated date, and approval status.
  • Add approve, reject, edit, delete, and merge flows for duplicate or conflicting facts.
  • Add change-probability review when new information updates an existing attribute.
  • Include commands and uploaded content as traceable Persona sources.

Testing:

  • Add a typed Persona fact and a voice Persona fact.
  • Verify duplicate facts merge instead of repeating.
  • Verify a changed fact creates an approval prompt before overwriting existing data.
  • Verify deleted/rejected facts disappear from Search and Persona summaries.

3. Mac Control Suite​

Goal: make the Mac Link app a complete trusted control surface for the connected Mac.

Implementation:

  • Add active app, window list, browser tab/title, clipboard, file search, screen list, and terminal status cards.
  • Add safe quick actions for focus app, paste clipboard, copy selected text, open URL, run allowlisted command, and highlight active screen.
  • Add server capability/version checks so the phone can explain when the Mac server is outdated.
  • Keep terminal, mouse, keyboard, and screen preview behaviors under the same Mac Link capability model.

Testing:

  • Start the Mac Link server and verify status/capabilities load.
  • Test active browser/window query, clipboard read/write, file search, and screen focus.
  • Restart an older server and verify VOXos shows an upgrade/capability warning instead of failing silently.

4. Smart Daily Brief​

Goal: turn the dashboard into a compact daily operating view.

Implementation:

  • Combine weather/location, next calendar events, travel watch, important mail, reminders, receipts, calls, health, provider status, and Persona approvals.
  • Add source chips so users can see where every brief item came from.
  • Add refresh and explain actions, using local LLM when available and deterministic fallback otherwise.

Testing:

  • Refresh with Google/Composio connected and disconnected.
  • Verify missing providers show setup guidance instead of empty cards.
  • Verify local LLM failure still produces a useful deterministic brief.

5. Local File Knowledge Base​

Goal: make selected local files, receipts, notes, music, and messages searchable and explainable.

Implementation:

  • Extend indexing metadata with source type, path, filename, MIME type, size, dates, tags, extracted text, and source permissions.
  • Add incremental reindexing and stale-source detection.
  • Add semantic summary hooks for local LLM when available.
  • Add export/delete controls for indexed metadata.

Testing:

  • Add a Files source and verify paths appear in search.
  • Reindex after changing a source and verify stale rows update.
  • Search across files, receipts, and notes.
  • Verify deleting a source removes its indexed rows.

5A. Hugging Face Hub Worker Jobs​

Goal: keep Hugging Face bucket list/tree sync metadata-only on device, and use explicit worker or Mac Link transfer only after the user chooses a download, upload, mirror, or large object operation.

Implementation:

  • Add queued Mac Link jobs for large bucket object upload/download/mirror and Git/Xet repository clone/push flows.
  • Add dry-run previews for destructive bucket deletes and large object mirror operations.
  • Keep iPhone/iPad direct calls focused on auth, bucket list/file metadata, sync/change history, small creates, and status inspection.
  • Store job logs and artifact metadata locally with provenance.

Testing:

  • Refresh Hugging Face profile, repositories, datasets, Spaces, buckets, selected tree metadata, sync history, and recent file-change history from iPad.
  • Run a Mac Link dry-run for a large bucket object mirror and verify no remote object changes occur.
  • Run a small approved upload through Mac Link and verify the job log returns to VOXos.
  • Confirm deletes require explicit confirmation and show irreversible-operation language.

6. Automation Rules​

Goal: let users define local-first rules such as "when X happens, do Y".

Implementation:

  • Add a Rules app with triggers for time, location, provider status, call trigger, notification, calendar event, Mac Link state, and manual command.
  • Add actions for open app, refresh provider, ring call, run Mac action, create note, update Persona proposal, or show alert.
  • Persist rules in SQLite with enabled state, last run, next run, and failure reason.
  • Require confirmation for risky actions.

Testing:

  • Create a time-based rule and verify it fires while the app is active.
  • Create a remote call trigger rule and verify it rings only when enabled.
  • Verify disabled rules do not run.
  • Verify risky Mac/provider actions require approval.

7. Privacy And Permissions Center​

Goal: give users one place to understand and control what VOXos can access and store.

Implementation:

  • Add permission cards for location, microphone, speech, photos, camera, notifications, health, files, Face ID, Google, Composio, Hugging Face, and Mac Link.
  • Add local data inventory for SQLite tables, Keychain-backed secrets, model files, indexed metadata, scans, receipts, and cache files.
  • Add export/delete/reset controls with confirmations.
  • Add plain-language limits where iOS blocks access, such as third-party notifications, SMS, and Passwords/iCloud Keychain.

Testing:

  • Toggle or revoke iOS permissions and verify status updates.
  • Verify export/delete/reset prompts use VOXos-styled alerts.
  • Verify Keychain secrets are reported as present without exposing values.

8. Receipt And Expense Intelligence​

Goal: turn receipt scanning into a useful expense and borrowing/lending manager.

Implementation:

  • Add better scan review for merchant, amount, category, tax, date, payment method, borrowed/lent/reimbursable state, and notes.
  • Add monthly reports, category charts, merchant summaries, and outstanding borrowed/lent lists.
  • Add search and Smart Brief integration for receipts and spending.
  • Add multi-photo receipt support where one expense has several images.

Testing:

  • Scan or import receipt photos and verify OCR prefill can be corrected.
  • Add spending, borrowed, lent, and reimbursable rows.
  • Verify monthly totals and category reports update.
  • Verify receipt rows appear in Search and Smart Brief.

Execution Rules​

  • Start from latest main.
  • Use a feature branch per task.
  • Build the Xcode app before installing.
  • Install on the connected iPhone when a build succeeds.
  • Create a PR, squash merge it, and fast-forward local main.
  • Update docs/FEATURES.md, docs/TESTING.md, and this plan whenever behavior changes.
