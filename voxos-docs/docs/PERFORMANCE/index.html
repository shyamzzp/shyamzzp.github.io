VOXos Performance Baseline

Purpose​

Phase 6 starts with measurement before tuning. The app now logs repeatable performance breadcrumbs through the com.shyamzzp.voxos / performance logger so cold launch, dashboard readiness, SQLite migration, provider refresh, navigation, and voice capture can be measured on a physical iPhone.

Current Instrumentation​

Use Console.app or Xcode device logs and filter for:

subsystem:com.shyamzzp.voxos category:performance

Logged points:

  • app_entry: SwiftUI app entry.
  • sqlite_open: local SQLite open duration.
  • sqlite_migration: migration duration and active schema version.
  • app_model_init: AppModel initialization duration with local row counts for facts, files, and receipts.
  • dashboard_first_render: first visible dashboard render.
  • dashboard_task_start and dashboard_task_ready: first async dashboard setup pass.
  • local_context_refresh: local context/weather/LLM explanation refresh duration.
  • provider_health_refresh: provider health refresh duration.
  • provider_refresh_composio_accounts: Composio connected-account refresh duration.
  • provider_refresh_schedule: Google/Composio schedule refresh duration and loaded event count.
  • provider_refresh_travel: Gmail/Composio travel refresh duration and loaded item count.
  • provider_refresh_google_photos: Google Photos metadata refresh duration and loaded item count.
  • global_search_compute: global search result snapshot computation duration, mode, term count, and result count.
  • local_llm_response and local_llm_optional_response: local GGUF readiness/generation duration, model name, output size, or failure reason.
  • detail_navigation: dashboard detail-page navigation duration by feature.
  • voice_press_start: long-press voice start marker.
  • voice_capture_start: microphone/listener startup duration.
  • voice_finalize_transcript: release-to-transcript duration with transcript character count.
  • parakeet_recording_start: local voice recording startup duration after microphone permission.
  • parakeet_prewarm: background model prewarm duration while the user is speaking.
  • parakeet_asset_validation: bundled/cached Parakeet asset validation duration.
  • parakeet_model_load: FluidAudio model load duration.
  • parakeet_manager_transcribe and parakeet_transcribe: local transcription duration and transcript character count.
  • apple_speech_fallback: fallback transcription duration when Parakeet is slow or unavailable.
  • parakeet_finalize: release-to-final-result duration for local voice.
  • receipt_image_prepare: receipt image normalization/storage preparation duration with stored byte count.
  • receipt_ocr_scan: Vision OCR scan duration and extracted character count.
  • intent_deterministic_route: cheap local command-router duration and hit/miss state.
  • intent_structured_route: local/cloud LLM JSON classifier duration when deterministic routing misses.
  • voice_chat_fallback: conversational fallback duration and selected provider.
  • widget_snapshot_publish: App Group snapshot publish path, including skipped unchanged snapshots.

Each log line includes elapsed app time and approximate resident memory when iOS exposes it.

SQLite Index Baseline​

Phase 6 added baseline indexes for high-traffic local tables:

  • notes(created_at)
  • persona_facts(attribute, created_at) and pending Persona review creation time
  • app_notifications(received_at)
  • command_audit(created_at) and command_audit(action, target, created_at)
  • indexed_messages(created_at) and indexed_messages(source, indexed_at)
  • indexed_files(is_stale, indexed_at), indexed_files(type, indexed_at), and indexed_files(source_type, indexed_at)
  • status_checks(created_at)
  • local_music_tracks(type, indexed_at)
  • local_models(kind, imported_at)
  • google_accounts(updated_at)
  • schedule_events(start_at) and schedule_events(source, account_email, start_at)
  • travel_info(start_at) and travel_info(source, account_email, indexed_at)
  • composio_results(toolkit, created_at)
  • receipts(receipt_date) and receipts(merchant, amount, receipt_date)

Use EXPLAIN QUERY PLAN against these paths before adding heavier search/indexing work.

Optimization Matrix​

  • Physical iPhone with existing local data.
  • Simulator with clean install.
  • Offline network.
  • Missing providers and expired credentials.
  • Local model absent and present.
  • Large receipt images.
  • Large terminal output.
  • Many Persona facts.
  • Many indexed file/search sources.
  • Widget snapshot enabled and disabled.

Phase 6 Order​

  1. Capture baseline numbers with the current instrumentation.
  2. Add missing timing around local LLM, global search, OCR, terminal capture, provider fetches, and widget writes.
  3. Tune startup and dashboard first render by deferring non-critical work. Local context/weather refresh now waits briefly until after dashboard_task_ready so first render and launch route handling are not competing with location/network work.
  4. Tune voice latency and deterministic intent routing. Obvious open/search/setup commands should use intent_deterministic_route; ambiguous commands should be the only common path reaching intent_structured_route.
  5. Add SQLite query timing, indexes, and query-plan checks.
  6. Reduce memory pressure for images, models, previews, and large terminal/search results.
  7. Audit timers and active/inactive policies for battery-sensitive loops.

Widget Update Policy​

Widget snapshots now compute a compact signature from the visible widget fields and selected app statuses. Non-manual publishes skip App Group writes and WidgetKit reloads when the signature is unchanged within 30 seconds. Manual publishes still force a write so Settings can be used for verification.

Large Capture and Indexing Policy​

  • Terminal refresh hashes the full Mac capture and skips publishing when unchanged, which reduces flicker and broad SwiftUI invalidation during fast refresh modes.
  • Terminal output stored in app state is capped to the newest mobile-visible window, while the full canonical terminal buffer remains on the Mac/tmux side.
  • File indexing and local music scans check cancellation during directory walks and update progress every bounded batch instead of publishing per-file UI state.
  • Files/Music detail surfaces show scanned/indexed progress so large folders are visibly active without forcing the dashboard to reload the full result list until the scan completes.
  • Global search now computes a cached, measured result snapshot when the query changes instead of rebuilding every result during unrelated SwiftUI redraws. It uses token matching for multi-word queries and caps the mobile result set to the first 80 ranked rows.

Voice Model Policy​

  • Parakeet asset validation metadata is cached per model directory after the required Core ML bundles and vocabulary pass size/existence checks.
  • The first Parakeet use can still pay model load cost, but recording now starts immediately and prewarm runs while the user is speaking.
  • If Parakeet exceeds the timeout or fails, Apple Speech fallback is timed separately so slow local model behavior can be diagnosed without hiding fallback latency.
  • Local GGUF responses now log full generation duration, and prompts are bounded before inference so accidental long provider/history text does not consume the entire local context window.

Receipt Image Policy​

  • Scanned receipts store a bounded JPEG copy for local review instead of retaining camera-resolution data in SQLite.
  • OCR uses a separate bounded working image so Vision can read full receipts without forcing the UI to hold the original capture size.
  • Receipt rows and preview sheets downsample image data with ImageIO before creating UIImage instances, keeping list scrolling and preview memory predictable.

Provider Refresh Policy​

  • Composio accounts, Schedule, Travel, and Google Photos refreshes use short in-memory exponential backoff after provider failures so repeated taps or queued retries do not hammer unavailable APIs.
  • Backoff is cleared on successful refresh and leaves existing cached rows visible.
  • Google Calendar, Gmail travel search, Google Photos, Composio Calendar, and GitHub/Composio paths already request bounded metadata/result sets rather than full provider payloads.

Battery and Lifecycle Policy​

  • Scene activation now flows through one AppModel lifecycle path so wake-word monitoring, notification monitoring, remote call trigger polling, automation rules, and Mac screen preview have one active/inactive policy.
  • When the app becomes inactive, wake-word monitoring, remote call trigger polling, automation rule polling, and live Mac screen preview stop. Status-call notifications remain scheduled through iOS notifications so lock-screen check-ins still work.
  • If Mac screen preview was running before the app became inactive, it resumes when the scene becomes active again.
  • Terminal auto-refresh is gated to the active scene, an open terminal full-screen session, an enabled Mac Link, no active keyboard editing, and the configured refresh interval. This keeps the 0.35s live timer from fetching while the terminal panel is closed or the app is inactive.
