Composio Integration

VOXos now has a Composio setup surface for connector/toolkit access.

Surfaces​

  • Dashboard grid item: Composio.
  • Account Integrations sheet: Composio row with inline setup details.
  • Voice commands: phrases mentioning Composio, connectors, or toolkits open the Composio detail page.

Stored Settings​

The current prototype persists these values in local SQLite:

  • composio_api_key
  • composio_user_id
  • composio_default_toolkit
  • composio_base_url

Default API base URL:

https://backend.composio.dev/api/v3.1

Current Behavior​

  • The user can enter the Composio API key and VOXos user ID.
  • Opening the Composio detail page automatically loads connected accounts for the configured VOXos user ID.
  • Active services are shown as tappable rows. Selecting a service opens a compact voice-first service panel on the same detail page.
  • Composio service interaction is voice-first: the user holds the Voice control, speaks to the active service, and VOXos transcribes the request into an intent proposal before any service call runs.
  • Intent proposals are classified as information, filter, or action requests. VOXos shows the heard transcript separately from the rephrased prompt that will be sent to the service, and the user approves that interpreted prompt before Composio receives the request.
  • The VOXos will ask prompt is now rewritten by the configured LLM instead of only string-prefixing the transcript. VOXos tries the selected local GGUF model first, then uses the configured cloud provider when cloud mode is active. If rephrasing fails, VOXos falls back to the deterministic service-specific prompt.
  • Gmail service chat is wired to Composio tool execution for fetching mail context through the approved Gmail connection.
  • Gmail voice intents are service-aware. For example, "filter mails that have attachments" is rephrased as a Gmail/GMAIL_FETCH_EMAILS request that focuses on Gmail attachment concepts such as has:attachment, filename, MIME parts, sender, subject, date, labels, and snippets before the user approves it.
  • Gmail responses are parsed into structured mail rows when the Composio response includes message-like fields such as sender, subject, snippet, and date. VOXos shows a compact preview list instead of raw JSON.
  • Tapping a Gmail row expands it inline and shows the detailed fields returned by Composio for that specific mail, including IDs, thread fields, recipients, labels, body/preview fields, and any other available payload keys.
  • Follow-up Gmail prompts such as filter transactions, show receipts, or keyword searches run against the current structured result set first, so the user can refine the visible list without immediately making another Composio call.
  • Non-Gmail services now use the generic service-result renderer. Safe read-only tool mappings currently include Google Drive through a Drive find/search action and GitHub through GITHUB_LIST_REPOSITORIES_FOR_THE_AUTHENTICATED_USER; unsupported toolkits show a setup card explaining that VOXos needs an explicit read-only tool mapping before execution.
  • GitHub service chat lists accessible repositories with deterministic Composio arguments, paginates up to the current safety cap, and renders compact repository rows with full name, visibility/description, activity dates, branch, language, stars, forks, and URL fields when returned.
  • GitHub follow-up prompts that mention commits, history, SHA, or changes now route to GITHUB_LIST_COMMITS instead of repeating repository listing. If the user does not name an owner/repo, VOXos uses the top visible repository from the current GitHub result set and renders commit rows.
  • The Composio detail page auto-scrolls to the latest result/conversation state when service messages or result lists change.
  • Composio service results are saved to local SQLite in composio_results with toolkit, connected account ID, prompt, title, summary, raw payload, and timestamp. The Composio detail page shows recent local result history when there is no active result list.
  • Other service panels open the selected service and show that a toolkit-specific tool mapping is still needed.

Next Step​

Add more toolkit-specific safe read mappings, then add schema inspection before any write/destructive action. The command flow already asks for user confirmation before execution; the next layer should inspect the Composio action schema first, block unsupported destructive actions, render service-specific rows/cards instead of raw payloads, and store action summaries locally with provenance.

Security Notes​

  • Token storage: the Composio project API key is stored in Keychain; VOXos user ID, toolkit, base URL, and result summaries stay in local SQLite.
  • Outbound data: approved prompts, interpreted tool arguments, and selected connected-account context are sent to Composio.
  • Destructive actions: unsupported write/delete/send actions are blocked until VOXos has an explicit safe mapping.
  • Confirmation: VOXos shows the heard transcript and rephrased intent before a Composio tool call runs.
  • Revoke path: remove connected accounts/toolkits in Composio, rotate the Composio key, and clear the key in VOXos.
