VOXOS Testing

Core Reliability Checklist​

Run this checklist before adding another major grid app:

  • Generate a feature verification checklist before starting implementation and save it beside the phase plan:
scripts/generate_feature_verification_checklist.py .planning/phases/<phase-dir>/<phase>-<plan>-PLAN.md --output .planning/phases/<phase-dir>/<phase>-<plan>-VERIFICATION.md

The generated checklist must cover build/install, core interaction, error handling, persistence, nearby regressions, plan-specific checks, and any automated validator that applies.

  • Validate the command intent golden dataset before changing command routing:
python3 scripts/validate_command_intent_golden_dataset.py
  • Build and install the native Xcode app on the connected iPhone.
  • For optimization changes, open Console.app or Xcode device logs and filter subsystem:com.shyamzzp.voxos category:performance. Record before/after launch, dashboard, navigation, provider, SQLite, and voice timings in docs/PERFORMANCE.md.
  • Open Settings and verify System Health shows the expected setup gaps instead of hiding them.
  • In Settings > Storage, verify the SQLite schema version and migration status are visible. Existing installs should show a successful migration and a backup path under Application Support/VOXOS/Backups/ when a version upgrade happened.
  • In Settings > Widgets, tap Publish widget snapshot and verify widget health shows App Group read/write status, last publish/read time, selected app names, feature count, and a concise recovery message if no snapshot exists. In Debug builds, confirm Data Inspector shows the same widget health without reading SQLite from the widget extension.
  • In a Debug build, open Data Inspector and verify SQLite table counts, latest safe row summaries, migration status, Keychain presence flags, and widget snapshot health are visible. Confirm it does not show raw API keys, OAuth tokens, card numbers, CVV, image data, OCR text, message bodies, notes, or provider payloads. In a Release build, Data Inspector should not appear in the dashboard grid.
  • In a Debug build, open a few feature routes, run one voice/text command, and trigger one provider setup/error path. Refresh Data Inspector and verify Recent Breadcrumbs shows redacted route, command, provider, or error entries. Breadcrumbs should be capped to the recent rolling buffer and must not include raw tokens, full transcripts, provider payloads, card data, or message bodies.
  • Confirm previously connected Google/Composio/Hugging Face/Mac Link settings and local data still appear after reinstall. API keys, OAuth tokens, and Hugging Face access tokens should remain loaded through iOS Keychain, not plaintext SQLite.
  • Open Hugging Face, tap Refresh Hub, and verify profile/repo/bucket metadata loads when the token has scope. Set optional repo and bucket defaults and verify bucket list rows, repo-tree/bucket-file metadata rows, sync history, and recent file-change history appear from local cache without downloading file contents.
  • On iPad, scroll the dashboard grid quickly and verify scroll tick feedback does not fire, the grid remains responsive, and detail intro information copy appears in neutral grey.
  • Open one dashboard detail page, scroll it, then use the configured swipe direction and back button to return to the main dashboard.
  • Open Terminal with a Mac Link session, leave the full-screen terminal open for several auto-refresh intervals, and verify unchanged output does not visibly flicker, capture failures preserve the last visible output, and the status line shows the latest refresh age.
  • Long-press Voice, confirm haptic feedback starts immediately, speak a command such as open music, release, and verify it opens the correct detail page or shows a VOXos-styled setup alert.
  • Open Voice after routing a command and verify Command history shows the latest source, intent, target, provider, risk, and relative timestamp.
  • Open Voice after routing successful, setup-needed, cancelled, and risky commands. Verify Action receipts show Done/Needs attention/Cancelled states, pending approvals can be confirmed or cancelled without reopening the original app, and follow-up prompts from Briefing/Search land in Voice without bypassing existing safety gates.
  • Open Voice, fill one or more Daily standup fields, review the approval preview, then tap Approve and save. Verify a local Note and Status Check are created with source Daily standup, the draft clears, and an Action receipt appears.
  • Open Briefing and use row-level Why, First, Changed, and Later actions. Verify each opens Voice with the selected briefing title/detail/source/priority context and does not bypass confirmation behavior for reminder-like actions.
  • Start a long-running operation such as Files indexing, Music scanning, Schedule/Travel refresh, Terminal capture, Mac preview, or model download. Verify the dashboard shows a compact Live status row while active, hides it when inactive, and opens the relevant app when tapped.
  • Open Voice and type change receipt category to travel with at least one receipt saved. Verify VOXos shows an Ask-to-edit card with before/after values, does not mutate before approval, then updates the latest receipt and creates an action receipt after Approve edit. Repeat with set Mac refresh to 2 seconds and verify Mac Link terminal refresh changes only after approval.
  • Open Voice and test set theme to graphite, switch to dark mode, set voice engine to cloud, and use Apple Speech. Verify each command shows an Ask-to-edit card with the Settings path and old/new values, saves only after approval, and creates an action receipt.
  • Open Search, enter a broad query, then use Voice/Search follow-ups such as only receipts, exclude stale files, and last month. Verify active filter chips appear, results narrow, the query remains intact, and Clear removes all filters.
  • Open Messages, save a local draft with recipient/channel/subject/source/body, and verify it appears under Drafts with Nothing sent context. Then type or speak draft a message to Alex saying I will be late, verify one new local draft appears, Search can find it with Drafts / Local SQLite provenance, and no provider send or external handoff occurs.
  • Open Schedule with at least one loaded event, open the event detail sheet, and tap What is this, Prepare, Related, and Talking points. Verify each action closes the sheet, opens Voice, includes event source/account/provider context in the routed prompt, and returns through the normal Voice/action receipt path.
  • Open Persona and verify the Response style card shows the enabled state, approved fact count, and pending-facts-excluded note. Test explain persona style, reset persona style, and enable persona style in Voice; verify each updates or explains the setting locally and adds an action receipt. With the setting off, normal Voice chat should not append Persona facts to LLM instructions.
  • Open Travel with at least one saved item, open a row, and tap Next ticket, PNR, Missing, and Checklist. Verify each closes the sheet, opens Voice, includes travel metadata/source mail/missing-field context, and does not perform provider writes.
  • Open Receipts and tap Spend, Food, Borrowed, and Reimburse in the Ask receipts panel. Verify Voice answers from local receipt rows only. Type change receipt category to travel and verify it still shows the Ask-to-edit approval card before mutating the latest receipt.
  • In Voice, ask summarize terminal output, explain terminal error, and refresh terminal capture. Verify Terminal opens, responses use the current captured output, refresh runs through the existing Mac Link terminal bridge, and command audit rows show provider Mac Link. Ask send terminal command pwd and verify a confirmation card appears before the command is sent.
  • In Voice, ask Mac Link status, read Mac clipboard, show Mac screen preview, pause Mac preview, focus Mac display under mouse, find Mac file README, and what browser is open on my Mac. Verify disabled Mac Link shows setup guidance, blocked capabilities show concise recovery text, allowed actions route through the existing Mac Link helpers, and command audit rows show provider Mac Link.
  • Open Focus and toggle Quiet mode on/off. In Voice, test quiet mode on, mute provider errors, brief me later, and quiet mode off; verify the Focus detail reflects the rule, the setting survives relaunch, command audit records the update, and ambient status whispers are reduced while quiet mode is enabled.
  • Load a Composio result set, then use Voice commands such as summarize current provider, filter current provider transactions, and clear provider session. Verify the visible result context is reused, filters do not trigger provider writes, and command audit records the provider-session actions. Repeat summarize provider session with Google Photos metadata loaded.
  • Trigger or simulate an error in Mac Link, Terminal, Google, Composio, local model, microphone, or Files. Ask Voice what went wrong or fix this. Verify VOXos opens the likely feature, gives one concise next step instead of raw JSON, and logs a redacted recover command audit row.
  • In Voice, ask read it to me, read my brief, or driving summary. Verify VOXos opens Voice, returns a compact schedule/travel/provider/briefing summary, avoids raw sensitive payload details, and records a Read-it-to-me command audit row.
  • Open Focus and enable Minimal home mode. Verify the dashboard grid shows only Voice, Briefing, Focus, Search, and Settings, then disable it and verify the full grid returns with existing data intact. Repeat with Voice commands minimal home on and minimal home off.
  • Open Search, enter terms that match Files, Notes, Receipts, and Commands. Verify each row shows compact provenance. Tap the info control and confirm provider, source table, indexed timestamp, permission scope, and freshness/stale/cache status are visible and truncate cleanly for long values.
  • Open Briefing and verify every row shows a compact priority score and reasons such as pinned, urgent, unresolved, local, provider, cached, or recent. Pin a related dashboard app, refresh briefing, and verify that pinned/urgent/unresolved items rank above stale low-value items.
  • Trigger a provider/network-like failure from Google, Composio, Hugging Face, Mac Link, or another provider surface and return to the dashboard. Verify the offline/provider-degraded banner appears without blocking the grid, lists local capabilities such as Notes, Persona, Receipts, Files, Local search, cached Schedule/Travel, and Local model, and opens Providers when tapped.
  • Open Files and Music, verify the iOS picker opens from the add-source buttons, and confirm selected file paths/tracks persist after relaunch.
  • Open Music, play a bundled or local MP3, lock the iPhone, and verify audio continues with Lock Screen / Control Center Now Playing controls.
  • Open Receipts, scan or import a full bill image in portrait and landscape orientation, verify the review sheet opens before the receipt is saved, inspect the photo, correct merchant/amount/category/type/OCR text if needed, save or discard, and confirm only saved scans appear in the receipt list. OCR Text should include lower receipt lines, the detected amount should match the actual total/payable amount in INR for formats like 1100/-, and item counters such as Total items 3 must not be used as the bill amount. Scan the same receipt again and verify VOXos shows a themed possible-duplicate warning; Discard should skip it, and Save Anyway should remain disabled until the user explicitly selects saving another copy. Legacy USD rows should display as INR; tap the saved thumbnail to preview the photo, use the OCR Text button beside Edit/Delete to review captured text, and open Edit to confirm the receipt editor uses VOXos-themed controls rather than native default chrome.
  • In Receipts, tap the Persona review action after saving a few receipts and verify VOXos opens Persona with proposed receipt-derived facts only in the approval queue. Approve one and discard one, then confirm approved facts persist while discarded suggestions do not.
  • Open Travel, connect Gmail directly or through Composio, choose a mail date range, and tap Refresh travel. Verify flight/train/hotel/cab-like messages produce rows with category, provider, reference, date, location/route, account, sender, and subject. Open a row and confirm the detail sheet shows structured fields, source provenance, message ID, compact raw metadata, and clear missing-field warnings when PNR/reference/date/location is not found.
  • In Persona, add a fact such as I work at Oracle, approve it, then add a conflicting fact such as I work at Apple. Verify VOXos stages a review row showing current value, proposed value, source, confidence, change probability, and suggested merge. Relaunch the app before approving to confirm the pending review survives. Approve one conflict and reject another; approved rows should update persona_facts, rejected rows should leave existing facts unchanged.
  • Open Composio, Google, Hugging Face, and Mac Link only after their System Health row is ready or clearly says what setup is missing.
  • Open Providers and verify the health center shows ready/setup/off counts, lists the top setup blockers with actionable text, and the refresh-all action updates Mac Link, Terminal, Composio, Google Photos, and notification permission status where configured.
  • Open Mac Link, run VOXOS_MAC_LINK_TOKEN="shared-token" VOXOS_MAC_LINK_ALLOW_TERMINAL=1 ./scripts/voxos_mac_link_server.py on the Mac from the repo root, enter the printed LAN URL and token in the guided pairing panel, tap Apply & Test, and verify the checklist reports URL, screen preview, mouse/keyboard, clipboard, and terminal readiness or clear setup blockers.
  • With the Mac Link server running, call GET /api/gateway/status or open the Gateway status URL in a browser and verify the response reports nodes, skills, channels, tool profiles, jobs, and capabilities without raw tokens. In VOXos, open Mac Link and refresh the Gateway panel; it should show live node, skill, channel, profile, and job counts. Search for gateway, openclaw, nodes, skills, and tool profiles; results should point to Daemons, Jobs, Device Fleet, Service Health, Browser Tasks, Memory Answer, or Automation Gallery rows with live Gateway readiness where available.
  • Open Pilotty after Mac Link is connected and verify it shows inherited Mac Link connection state without requiring Test Mac. Run the Pilotty help or smoke command and verify VOXos preflights the terminal bridge, sends to the selected tmux target, and refreshes terminal output after the command runs.
  • Open Terminal, select a Mac terminal target, and verify the safety profile selector is visible. In Read-only, ls should send and rm -rf test-folder should be blocked. In Confirm destructive, rm -rf test-folder should show the app-styled review row and should not send until Run once is tapped. Long-press the terminal mic and speak a destructive command to confirm voice uses the same gate.
  • Open Rules, add the sample rule, enter dry-run trigger text, and tap Dry Run on the rule. Verify the preview shows proposed action, target, risk, confirmation requirement, safety note, and No changes. Try enabling a disabled rule before dry-running it and confirm VOXos asks for dry-run first. Confirm the dry run does not create notes, start calls, navigate, or call providers.
  • In Providers, tap Export diagnostics, inspect the generated JSON preview/share sheet, and verify it includes app settings, provider readiness, recent command audit summaries, local data health, widget health, and redacted error summaries. Confirm it does not include raw API keys, OAuth tokens, card numbers, CVV, image data, OCR text, message bodies, notes, provider payloads, or full command transcripts.
  • Open representative deep links from Safari, Notes, or Shortcuts: voxos://feature/files, voxos://feature/composio, voxos://feature/terminal, voxos://settings/appearance, voxos://settings/voice, and voxos://feature/settings?section=google. Each should open the matching detail page, and Settings subsection links should highlight the requested subsection. Try an invalid link such as voxos://feature/not-real and verify VOXos shows a concise app-styled failure message.
  • In Providers, queue a health refresh, verify it appears in Sync queue, tap retry, and confirm it moves through running to completion or shows a concise failed state with retry count and next error.
  • If a provider request fails, verify VOXos shows a concise app-styled message instead of raw JSON or a native alert.

Dashboard Navigation Smoke Test​

Run:

./scripts/ui_smoke_dashboard_navigation.sh

What it covers:

  • Builds the native Xcode app for the booted iPhone 16 Pro simulator.
  • Installs and launches VOXOS.
  • Captures the detail page before the back action.
  • Taps the detail back control.
  • Captures the screen after the back action.

Expected result:

  • The after screenshot should show the main dashboard welcome section, not the detail header.
  • Screenshots are written to /tmp/voxos-ui-smoke-before-back.png and /tmp/voxos-ui-smoke-after-back.png.

Notes:

  • Synthetic clicking requires macOS Accessibility permission for the terminal running the script.
  • If the click is blocked, the script exits with setup guidance instead of reporting a false pass.

XCUITest Foundation​

The project includes VOXOSUITests with dashboard reliability coverage. Current smoke tests verify:

  • App launch starts on the main dashboard, not a restored detail page.
  • Detail back button returns to the dashboard.
  • Settings opens, shows System Health, reports Local database readiness, and exposes the Storage save action.
  • Terminal opens as an integrated terminal surface and shows either switchable terminal targets or Mac Link setup guidance.

If xcodebuild test reports no simulator destination, install the matching iOS simulator runtime for the active Xcode SDK from Xcode Settings > Components, then run:

SIM_ID="$(xcrun simctl list devices booted | awk -F '[()]' '/iPhone 16 Pro/ { print $2; exit }')"
xcodebuild test \
  -project voice_xcode/VOXOS.xcodeproj \
  -scheme VOXOS \
  -configuration Debug \
  -destination "id=$SIM_ID" \
  -only-testing:VOXOSUITests/DashboardNavigationUITests

The latest verified simulator run used iPhone 16 Pro simulator FF8A763F-291C-42CF-AEDE-55F41B784655 and passed 4 tests with 0 failures.
