Mac Link / KDE Connect Bridge

VOXos includes a Mac Link grid app for connecting the iPhone app to a trusted macOS companion server over LAN, VPN, or a secure HTTPS tunnel. The phone or iPad only needs internet access when a tunnel URL is configured.

iOS App​

  • Grid item: Mac Link
  • Settings stored locally in SQLite:
    • enabled state
    • Mac server URL
    • Mac display name
    • bridge mode
    • notes
  • Pairing token is stored through the existing secret storage path.
  • Test action checks GET /health, GET /status, GET /api/status, then the base URL.
  • The latest companion server reports server_version and capability flags for commands, KDE Connect, clipboard read/write, screen preview, mouse/keyboard, Terminal, web terminal, browser inspection, file search, and power actions. VOXos shows these flags in Mac Link so setup blockers are visible without trying each command manually.
  • The companion server also exposes GET /api/gateway/status and GET /gateway/status as a read-only VOXos Gateway contract. The payload reports trusted nodes, Mac-side skills, future channels, tool profiles, jobs, and capability flags. It does not expose tokens and does not run terminal, browser, provider, or Mac-control actions by itself.
  • Mac Link includes a guided pairing panel. Enter the server URL printed by the Mac script, enter the optional pairing token, tap Apply & Test, then review the checklist for server URL, token, screen preview, mouse/keyboard, clipboard, and terminal readiness.
  • Command action posts to POST /api/command, POST /command, POST /commands, then POST /kdeconnect/command.
  • Screen preview polls GET /api/screens / GET /screens for all active Mac displays while the Mac Link detail screen is open. The response includes each display's CoreGraphics x, y, width, and height, so VOXos can arrange previews like macOS Display Settings. It falls back to GET /api/screen, GET /screen, GET /api/preview, then GET /preview for older servers.
  • Mac Link has a Profile Sync panel scoped to the active Google account selected in VOXos Profile/top navigation. iPhone and iPad devices using the same active account can refresh, list, share, and adopt saved Mac endpoints without repeating manual URL/token setup.
  • Profile Sync now uses Supabase as the shared source of truth. VOXos exchanges the active Google ID token for a Supabase Auth session, then upserts/fetches rows from the voxos_profile_maclink_endpoints table. Local SQLite remains only a device cache.
  • Shared Mac endpoint rows include the pairing URL and token required by iPhone/iPad adoption. Run scripts/check_supabase_profile_sync.sh before using cross-device sync, and keep the table prefix as voxos_ so this Supabase database can host other apps.

Active Profile Sync​

The active Google account is the Mac Link namespace. Switching from one Google account to another switches the visible shared Mac endpoints and the settings that can be adopted.

Workflow:

  1. Connect Google from Profile/Settings and make the intended account active.
  2. Configure Settings > Supabase with the project URL, publishable key, and voxos_profile_maclink_endpoints.
  3. Tap Settings > Supabase > Check Supabase; it should report that the table is reachable and Google Auth is enabled.
  4. Pair Mac Link once by QR/manual URL, then tap Share this Mac in VOXos > Mac Link.
  5. On another iPhone or iPad signed into the same active Google account, open VOXos > Mac Link and tap Refresh profile.
  6. Choose the shared Mac endpoint and tap Use; VOXos applies the URL/token and tests the connection.

The macOS companion has a Profile Sync section where the same profile email can be entered and the profile endpoint payload can be copied. The companion does not currently perform Google OAuth itself; iPhone/iPad publish and refresh Supabase rows while the Mac companion keeps the server/tunnel running.

Development Server​

Run this on the Mac:

python3 scripts/voxos_mac_link_server.py

Optional token:

export VOXOS_MAC_LINK_TOKEN="shared-token"
python3 scripts/voxos_mac_link_server.py

Then open VOXos > Mac Link and set the URL to the LAN address printed by the script, for example:

http://192.168.1.20:8765

The guided pairing panel can save and test this in one step. If a capability is blocked, use the checklist text to enable the matching macOS permission or server flag, then restart the server and tap Apply & Test again.

Fresh macOS Permissions After Rebuild​

Debug Mac companion builds can leave stale macOS TCC entries when the app is rebuilt at a new path or with a new code signature. The companion now resets its own Accessibility and Screen Recording decisions once per fresh Debug executable, then requests Accessibility again on launch. macOS still requires the user to approve the prompt; the app and scripts can reset decisions but cannot grant permission silently.

Manual reset:

scripts/reset_voxos_maclink_privacy.sh

Fresh build, reset, and launch:

scripts/dev_maclink_fresh.sh

The macOS companion UI also includes Fresh Permission Reset in Server Control, plus menu bar actions for Accessibility, Screen Recording, and Reset Privacy Permissions.

If Mac Link is launched with python3 scripts/voxos_mac_link_server.py from Terminal or iTerm, macOS stores Accessibility and Screen Recording decisions under that terminal app instead of VOXOSMacLink.app. Reset those entries explicitly when needed:

scripts/reset_voxos_maclink_privacy.sh --terminal-hosts

Mac Link can also register itself with the VOXos Gateway runtime when the backend is running:

export VOXOS_GATEWAY_URL="http://127.0.0.1:8000"
export VOXOS_GATEWAY_TOKEN="shared-token"
python3 scripts/voxos_mac_link_server.py

The server posts its node ID, base URL, server version, tool profile, and capability list to POST /gateway/nodes/register. For diagnostics without registration, call GET /api/gateway/node. After registration, Mac Link sends periodic heartbeats to POST /gateway/nodes/{node_id}/heartbeat; set VOXOS_GATEWAY_HEARTBEAT_SECONDS to change the default 60-second interval.

For terminal access with a scannable QR payload on the local network:

brew install tmux qrencode
scripts/start_voxos_mac_link_terminal.sh

The script starts Mac Link with VOXOS_MAC_LINK_ALLOW_TERMINAL=1, prints the URL and token, and prints a QR code when qrencode is installed.

Pilotty And Browser Automation​

Pilotty and Agent Browser are grid apps that control Mac-side CLI tools through the existing Mac Link terminal bridge. They do not require a separate Mac connection once Mac Link is connected; their detail screens show the inherited Mac Link state and the selected terminal target.

Before sending a Pilotty or Agent Browser command, VOXos refreshes terminal bridge readiness if the connection looks stale, creates/selects the default tmux target when no target is selected, sends the command, then refreshes captured output again after the command has had time to run.

If commands stop working after server changes, relaunch the macOS companion or terminal server with terminal access enabled:

VOXOS_MAC_LINK_ALLOW_TERMINAL=1 python3 scripts/voxos_mac_link_server.py

Use scripts/start_voxos_mac_link_terminal.sh when you want the QR/token helper and tmux setup checks.

Internet Access​

Mac Link cannot reach a private 192.168.x.x Mac address from cellular, hotspot, hotel Wi-Fi, or another network unless there is a route back to the Mac. For internet access from anywhere, run the companion server behind a tunnel and paste the tunnel URL into VOXos.

Quick internet tunnel:

brew install cloudflared tmux qrencode
scripts/start_voxos_mac_link_internet.sh

The script starts the Mac Link server on 127.0.0.1, starts a Cloudflare quick tunnel, prints an HTTPS trycloudflare.com URL, prints a generated pairing token, and prints a QR code when qrencode is installed. In VOXos > Mac Link, tap the scanner icon beside Guided Mac Link setup to fill the URL and token from the QR code, then Apply & Test.

Quick tunnel URLs can change when the tunnel restarts. For a stable URL, use a named Cloudflare Tunnel, Tailscale Funnel, or another trusted reverse tunnel and point it at the local Mac Link server port.

Companion URL Stability​

The macOS companion caches the last local pairing address with a network signature made from the selected interface, IPv4 address, Wi-Fi name when macOS reports it, and port. Starting the server or copying the pairing payload keeps the same local URL while that signature is unchanged, then updates the local URL when the route, address, Wi-Fi identity, or port changes.

The companion pairing token is also persisted in local macOS settings. It remains stable across companion relaunches and server restarts, and changes only when Rotate Token is used. After rotating, restart the server and scan/copy the new pairing payload on the iPhone.

The companion also keeps the latest public tunnel URL in local settings and does not clear it while a new Cloudflare tunnel is starting. Cloudflare quick tunnel URLs can still change after a tunnel restart; use a named tunnel or custom hostname when the public URL must be guaranteed stable.

KDE Connect​

The development server checks for kdeconnect-cli. It currently supports:

  • ping
  • status
  • help
  • system info
  • battery
  • network / wifi
  • front app
  • running apps
  • what browser is open
  • what is the current browser tab
  • current browser URL
  • open app Safari
  • open url https://example.com
  • finder path
  • desktop files
  • downloads files
  • find file budget
  • screenshot
  • screen preview
  • mouse status
  • mouse move <dx> <dy>
  • mouse click
  • mouse right click
  • mouse double click
  • mouse scroll <amount>
  • volume
  • set volume 40
  • mute / unmute
  • clipboard status
  • set clipboard <text>
  • read clipboard only when the server is started with VOXOS_MAC_LINK_ALLOW_CLIPBOARD_READ=1
  • lock screen / sleep mac only when the server is started with VOXOS_MAC_LINK_ALLOW_POWER_ACTIONS=1
  • kde devices
  • kde refresh
  • kde ping <device-id>
  • kde share <device-id> <path-or-url>
  • Terminal bridge endpoints are available only when the server is started with VOXOS_MAC_LINK_ALLOW_TERMINAL=1; see docs/TERMINAL_INTEGRATION.md.

Browser inspection is read-only and uses AppleScript for supported browsers: Safari, Chrome, Edge, Brave, Arc, Vivaldi, and Opera. The first request can trigger macOS Automation permission prompts. If VOXos reports that macOS blocked inspection, open System Settings > Privacy & Security > Automation and allow the terminal app running the server to control System Events and the browser.

Clipboard read and power actions are disabled by default. Enable them explicitly with environment variables only on a trusted private network:

VOXOS_MAC_LINK_ALLOW_CLIPBOARD_READ=1 VOXOS_MAC_LINK_ALLOW_POWER_ACTIONS=1 python3 scripts/voxos_mac_link_server.py

The server intentionally does not expose arbitrary shell execution. Add new behavior as explicit allowlisted Mac-side commands before routing it to KDE Connect or macOS automation.

Phone Touchpad​

VOXos > Mac Link includes a phone touchpad panel:

  • drag inside the pad to move the Mac pointer
  • tap for left click
  • double tap for double click
  • use the buttons for left click, right click, scroll up, and scroll down

The touchpad posts to POST /api/mouse with explicit actions: move, click, double_click, right_click, and scroll.

Mouse control uses macOS CoreGraphics from the companion server. On first use, macOS may require Accessibility permission for VOXOSMacLink.app or for the terminal app that runs the script server. Enable it in System Settings > Privacy & Security > Accessibility, then restart the server if pointer movement is blocked.

Live Screen Preview​

VOXos > Mac Link shows compact live previews of all active connected Mac screens above the touchpad. The app refreshes previews while the Mac Link detail page is visible and pauses them when the page closes.

Dragging across any live preview sends throttled move_to events with the Mac display x and y coordinate from the finger position. A short tap sends click_at, so iPhone and iPad taps can directly activate items on the Mac screen while longer drags feel closer to a Zoom-style remote-control session.

The companion server serves all active displays from:

  • GET /api/screens
  • GET /screens

It also serves a single display or combined fallback frame from:

  • GET /api/screen
  • GET /screen
  • GET /api/preview
  • GET /preview

The server captures JPEG frames with macOS screencapture -C, which includes the mouse pointer where macOS permits cursor capture. It enumerates active displays through CoreGraphics, captures each display separately, downsizes only above the configured source width with sips, and returns frames with Cache-Control: no-store for single-image endpoints. The default max preview width is 4096 pixels so terminal text remains readable when the iPhone preview is zoomed. Override it when starting the server:

VOXOS_MAC_LINK_SCREEN_PREVIEW_MAX_WIDTH=5120 python3 scripts/voxos_mac_link_server.py

Screen preview requires macOS Screen Recording permission for VOXOSMacLink.app or for the terminal app that runs the script server. Enable it in System Settings > Privacy & Security > Screen Recording, then restart the server if the preview is blank or unavailable.

The iPhone app includes a full-screen preview button. Rotate the phone to landscape while that view is open for a wider remote-monitor view.

Both the compact and full-screen preview include zoom controls up to 500%. Full-screen mode also has a bottom control switcher:

  • Mouse mode: touchpad, left click, right click, scroll up, and scroll down.
  • Keyboard mode: multiline text entry sent to the Mac through POST /api/keyboard or POST /keyboard.

Keyboard input uses macOS System Events, so the terminal app running the server may need Automation and Accessibility permission.

Security Notes​

  • Token storage: the Mac Link pairing token is stored in Keychain; server URL and capability flags stay in local settings.
  • Outbound data: commands, clipboard text, file-search terms, mouse/keyboard events, and screen-preview requests are sent to the Mac server.
  • Destructive actions: power, file, keyboard, clipboard, and shell-adjacent commands can affect the Mac.
  • Confirmation: risky commands should be allowlisted on the server and confirmed in VOXos before execution.
  • Remote access: prefer VPN or HTTPS tunnel access over opening a router port directly, and never expose the server without a pairing token.
  • Revoke path: stop the Mac server, rotate VOXOS_MAC_LINK_TOKEN, or disable Mac Link in VOXos.
