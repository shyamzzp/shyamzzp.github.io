Terminal Integration

VOXos includes a Terminal dashboard app for controlling trusted macOS terminal sessions through the existing Mac Link companion server. The app opens as a terminal-like tmux session: output is shown in the main terminal surface, commands are typed at the prompt, and results appear in the same session on the Mac.

Architecture​

  • iPhone app: native VOXos Terminal grid item.
  • Transport: existing Mac Link URL and optional pairing token.
  • Mac server: scripts/voxos_mac_link_server.py.
  • Session backend: tmux, plus guarded native Terminal/iTerm target selection through macOS Automation.
  • Optional web terminal: ttyx, exposed as a URL after Mac Link validates the request.

The iPhone does not get direct unauthenticated shell access. It asks the Mac companion server to manage named tmux sessions, capture output, and send text into the selected tmux, Terminal, or iTerm target after Mac Link token validation.

Mac Setup​

Install tmux:

brew install tmux

Optional ttyx setup:

git clone https://github.com/risacher/ttyx.git
cd ttyx
npm install

Start the VOXos Mac Link server with terminal access explicitly enabled:

VOXOS_MAC_LINK_ALLOW_TERMINAL=1 python3 scripts/voxos_mac_link_server.py

Recommended development launcher:

./scripts/start_voxos_mac_link_terminal.sh

This runs the Mac Link server inside a persistent tmux session named voxos-mac-link-server, with terminal access enabled on port 8766, and writes logs to /tmp/voxos_mac_link_8766.log.

Optional token and ttyx URL:

VOXOS_MAC_LINK_TOKEN="shared-token" \
VOXOS_MAC_LINK_ALLOW_TERMINAL=1 \
VOXOS_MAC_LINK_TTYX_URL="http://192.168.1.20:3000" \
python3 scripts/voxos_mac_link_server.py

Server Endpoints​

  • GET /api/terminal/status: reports terminal bridge, tmux, node, ttyx, and default session readiness.
  • GET /api/terminal/sessions: lists tmux sessions.
  • POST /api/terminal/session: creates or attaches a named tmux session.
  • GET /api/terminal/capture?session=voxos: captures recent output from a tmux pane.
  • POST /api/terminal/send: sends text and optional enter key into the selected tmux session.
  • GET /api/terminal/instances: lists running Terminal and iTerm windows/tabs/panes by title and target ID.
  • GET /api/terminal/instance/capture?target_id=iterm:1:1:1: captures output from a native Terminal/iTerm target where AppleScript permits it.
  • GET /api/terminal/instance/focus?target_id=iterm:1:1:1: activates the selected native Terminal/iTerm window, tab, or pane on macOS before VOXos captures it.
  • POST /api/terminal/instance/send: sends text and optional enter key into a native Terminal/iTerm target where Automation and Accessibility permissions permit it.
  • POST /api/terminal/open: opens the selected tmux session in Terminal or iTerm on the Mac.
  • GET /api/terminal/url?session=voxos: returns the configured ttyx URL for an embedded terminal view.

Existing Terminal and iTerm Windows​

VOXos lists running Terminal windows/tabs and iTerm windows/tabs/panes through macOS Automation. The Terminal app shows these targets in the same switcher as tmux sessions, so the user can select a target and continue typing from the phone. tmux remains the most reliable option for long-running sessions and exact pane capture, but native Terminal/iTerm targets are supported when macOS grants the server Automation and Accessibility access.

If window listing or open-in-Terminal fails, allow the terminal app running the Mac Link server to control System Events, Terminal, and iTerm in System Settings > Privacy & Security > Automation. Opening and controlling terminal windows can also require Accessibility permission.

Security​

Terminal access is disabled by default. Enable it only on a trusted private network:

VOXOS_MAC_LINK_ALLOW_TERMINAL=1

Recommended:

  • Use VOXOS_MAC_LINK_TOKEN.
  • Do not expose the server to the public internet.
  • Use named tmux sessions instead of arbitrary shell execution.
  • Keep destructive shell commands behind explicit user confirmation in the app.

Terminal Safety Profiles​

The Terminal app includes a visible safety profile selector in the terminal session:

  • Read-only: permits inspection commands such as ls, pwd, cat, rg, grep, git status, git log, and git diff; blocks write/build/destructive commands.
  • Project-local: permits normal project commands, asks for confirmation when a command references parent, home, or absolute paths, and asks before destructive commands.
  • Build/test: permits read-only commands plus build and test commands such as xcodebuild, swift test, npm test, pytest, and make test; blocks destructive commands.
  • Confirm destructive: permits normal commands and asks before risky commands such as rm, sudo, chmod, kill, git reset, git clean, destructive Docker/Kubernetes operations, and shell redirection.

Typed, pasted, and terminal voice commands use the same safety gate before they are sent to the Mac. Terminal control keys such as Return, Tab, Ctrl-C, Delete, and Clear still send raw key events because they are not free-form shell commands.

Threat Notes​

  • Token storage: Terminal uses the Mac Link token; terminal target/session state stays in local SQLite.
  • Outbound data: typed, pasted, and spoken commands are sent to the selected tmux, Terminal, or iTerm target.
  • Destructive actions: shell commands can modify, delete, install, push, or expose Mac/project data.
  • Confirmation: safety profiles block read-only violations and require explicit approval for destructive commands when configured.
  • Revoke path: restart the Mac server without VOXOS_MAC_LINK_ALLOW_TERMINAL=1, stop the server, or rotate the Mac Link token.

iPhone Usage​

  1. Open VOXos > Mac Link and configure the Mac server URL/token.
  2. Open VOXos > Terminal.
  3. Tap Refresh.
  4. Select any listed tmux session, Terminal tab, or iTerm tab/pane.
  5. Create a new tmux session if you want a dedicated VOXos-controlled shell.
  6. Use Mirror on Mac to attach the selected tmux session in Terminal.
  7. If ttyx is running and VOXOS_MAC_LINK_TTYX_URL is configured, VOXos embeds the ttyx web terminal as the primary fully interactive PTY surface.
  8. If ttyx is unavailable, VOXos falls back to its capture/send terminal prompt. Type commands directly into that terminal prompt, use Refresh to recapture output, and Clear to clear the selected shell.

ttyx is the preferred path for exact terminal-style interaction. The direct Mac Link prompt remains as a fallback when ttyx is not installed or not reachable.

Terminal Screen Behavior​

  • Selecting a tmux, Terminal, or iTerm target opens a full-screen terminal session.
  • The session header includes Refresh, Close, refresh interval controls, and quick terminal keys.
  • The fallback prompt keeps the command input visible above the iPhone keyboard.
  • The terminal output scrolls independently from the main Terminal app page.
  • Auto-refresh runs only while the full-screen terminal session is open.
  • Auto-refresh pauses while the fallback command prompt is focused or the keyboard is open.
  • The refresh interval is configurable from the full-screen terminal session and persists locally.
  • The last selected tmux, Terminal, or iTerm target persists in local SQLite and is restored on app launch.
  • Reconnect refreshes the restored target and recaptures its latest output without requiring the user to pick it again.

Quick terminal keys:

  • Return sends an actual terminal return key.
  • Tab sends terminal tab completion.
  • Ctrl-C sends an interrupt.
  • Clear sends clear and return.
  • Delete sends terminal delete, or line clear when Control is active.
