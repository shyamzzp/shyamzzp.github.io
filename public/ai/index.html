Module 01 · Foundations
The AI Spectrum:
ML → Agentic AI
Understanding the five concentric layers of AI capability — from classical machine learning all the way to fully autonomous agentic systems that automate entire business processes.
The 5 Layers Explained

Think of AI capability as concentric circles. Each outer ring includes and extends everything inside it. You can't have Agentic AI without understanding Gen AI; you can't have Gen AI without Neural Networks. Each layer adds a new category of capability.

01
AI & ML — "Turn your data into decisions"
The foundation. Classical machine learning uses algorithms to find patterns in structured data and make predictions. Includes Supervised Learning (labeled data → predictions), Unsupervised Learning (find hidden clusters), and Reinforcement Learning (reward-based trial and error). Example: Gmail spam filter, Netflix recommendation engine, credit score prediction.
02
Neural Networks — "Complex business pattern detection"
Deep learning architectures that model complex non-linear relationships. Key architectures: CNNs (images — used in medical imaging, self-driving cars), RNNs/LSTMs (sequences — used in time-series, speech), Transformers (attention mechanism — the basis of all modern LLMs). Example: facial recognition, fraud detection in transaction sequences.
03
Gen AI — "Generate content and code at scale"
Large Language Models + generation capabilities. Takes Transformers and scales them to billions of parameters trained on the internet. Key capabilities: text, code, image, audio generation. Key technique: RAG (grounds generation in external knowledge), Prompt Engineering (controls behavior without retraining). Example: ChatGPT writing an email, DALL-E generating product images, GitHub Copilot suggesting code.
04
AI Agents — "Execute complex tasks autonomously"
LLMs given tools, memory, and a planning loop. An agent can perceive its environment, reason about what to do, and act using tools (search, APIs, databases). Key additions: Tool Use, Memory (short/long term), Goal Decomposition, Human-in-the-Loop oversight. Example: Cursor (AI coding assistant), Perplexity (search + synthesize), customer support bots that actually take actions.
05
Agentic AI — "Automate entire processes with AI"
Multiple AI agents coordinating to complete long-horizon, complex goals. Adds: Multi-agent Collaboration, State Persistence across sessions, Self-improving loops, Governance & Guardrails, Rollback Mechanisms. Example: A supply chain management system where one agent monitors inventory, another negotiates with suppliers, another updates ERP — all without human intervention.

Module 01 · Ecosystem
Open-Source AI
Ecosystem Map
The complete landscape of open-source tools organized by function — from raw data sources to production deployment. Understanding this map lets you pick the right tool for each layer of your AI system.
The 10 Ecosystem Layers
Layer 1
Data Sources & Knowledge Stores
Where your AI gets its knowledge. Sources: HuggingFace Datasets, Common Crawl (the internet), Wikipedia dumps. Used for: Training, benchmarking, and RAG indexing. Key insight: the quality of your data source directly determines the quality of your model's knowledge.
Layer 2
Open-Source LLMs
Models: Llama 3 (Meta), Mistral, Falcon, Gemma (Google), Qwen (Alibaba). Use cases: Text generation, chatbots, reasoning. Advantage: Full transparency + customizable for enterprise. Run locally or on your own GPU cluster. No API fees, full data privacy.
Layer 3
Embedding Models
Tools: Sentence Transformers, E5, BGE, Instructor models. Use cases: Search, similarity scoring, clustering, RAG pipelines. These models convert text into dense vectors (numbers) that capture meaning. Semantically similar text gets vectors close together in space.
Layer 4
Vector Databases
Projects: Chroma, Weaviate, Milvus, Pinecone (hybrid), Qdrant. Role: Fast semantic search + memory storage for AI agents. Features: Indexing, filtering, metadata search. The backbone of any RAG system — stores embeddings and retrieves the most relevant ones at query time.
Layer 5
Prompting & Fine-Tuning
Libraries: PEFT, LoRA, QLoRA, Axolotl, Alpaca-Lite. Purpose: Lightweight fine-tuning of LLMs without retraining from scratch. Goal: Fast, scalable model serving. LoRA adds tiny trainable matrices to a frozen model — 90% cheaper than full fine-tuning while achieving similar results.
Layer 6
Deployment & Serving
Tools: FastAPI, Triton Inference Server, VLLM, HuggingFace Inference. Goal: Fast, scalable model serving. VLLM uses PagedAttention to serve LLMs 24x faster. Triton handles batching and GPU optimization. These tools are what turn a trained model into a production API.
Layer 7
RAG & Search Frameworks
Systems: Haystack, LanceDB, ColBERT, LlamaIndex RAG. Role: Enhances LLM accuracy with knowledge retrieval. Importance: Critical for production-grade AI. A RAG framework orchestrates the embedding, storage, retrieval, and augmentation pipeline automatically.
Layer 8
Agent & Orchestration Frameworks
Tools: LangChain, LlamaIndex, CrewAI, AutoGen. Purpose: Planning, memory, RAG, tool-use, reasoning. Strength: Scalable, flexible, GPU-optimized workflows. These frameworks wire together LLMs + tools + memory into a coherent agent loop so you don't have to build the plumbing yourself.
Layer 9
Evaluation & Guardrails
Tools: DeepEval, Ragas, LangSmith, Guardrails AI. Use case: Hallucination detection, prompt testing, safety filters. You can't trust what you don't measure. Ragas provides automated evaluation metrics for RAG systems (context recall, faithfulness, relevance).
Layer 10
MLOps & Model Management
Platforms: MLflow, BentoML, Kubeflow, Ray Serve, DVC. Functions: Experiment tracking, versioning, deployment, monitoring. Value: Reproducibility + scalable pipelines. MLOps is what separates a notebook experiment from a production system.

Module 02 · Agent Concepts
3 Types of
AI Workflows
Not every AI system is an "agent." Understanding these three distinct workflow types is foundational — it determines how you design, build, and evaluate any AI system.
Type 1: Non-Agentic Workflow

Definition: A single, stateless LLM call. User provides input, LLM processes it, output is delivered. No planning, no tools, no feedback loop. The simplest possible AI interaction.

Characteristics: No memory between calls, no tools, no self-evaluation. The model has no ability to try again if its answer is wrong. Every prompt is independent.

Real Example
Asking ChatGPT a Single Question
You type: "What is the capital of France?" ChatGPT responds: "Paris." That's it. No tool was called. No plan was made. No memory will persist. This is a Non-Agentic Workflow. Use it for: simple Q&A, one-off summarization, translation, single-turn content generation.
Type 2: Agentic Workflow

Definition: A structured process where an LLM follows a pre-defined pipeline — with tools, evaluation, and iteration — but the overall workflow logic is programmed by a human. The LLM doesn't set its own goals; it executes a fixed agentic process.

Key difference from Type 3: The workflow structure is predefined. The agent doesn't autonomously decide what to do next — it follows a script that includes LLM calls.

Real Example
GitHub Copilot Writing Code
You describe a function. Copilot: (1) Generates a code plan, (2) Writes the code, (3) Evaluates it for syntax errors, (4) Suggests a revised version if issues found. The workflow is agentic (it uses tools + evaluation), but the overall structure is fixed — you didn't give it a broad goal to pursue autonomously.
Type 3: AI Agent

Definition: You give the AI a high-level goal. It autonomously: understands its environment, defines sub-goals, plans a path, executes actions using tools, evaluates outcomes, and replans if needed — without step-by-step human instructions.

Critical distinction: The agent decides what to do, not just how to do it. It can change course, use unexpected tools, and handle situations not explicitly programmed.

Module 02 · Agent Concepts
5 Levels of
AI Agents
HuggingFace's progressive framework for classifying AI agents by how much control the LLM has over the program flow. Level 1 is a simple processor; Level 5 is fully autonomous code-executing AI.
The 5 Levels in Detail
L1
Simple Processor — Model has no impact on program flow
The LLM receives an input and produces an output. The program doesn't change based on the model's response. All logic is hardcoded by the developer. Workflow: Input → LLM → Output (done). Example: A sentiment analysis API. You send text, get "positive/negative/neutral" back. The program flow is fixed regardless of what the model returns.
L2
Router — Model determines basic program flow
The LLM's output determines which of several predefined paths the program takes. The model acts as a classifier/router. Workflow: Input → LLM → [Branch A or Branch B] → Output. Example: A customer service bot that uses an LLM to classify whether your query is "billing", "technical", or "account" — then routes to the right handler. The routing decision is made by the model.
L3
Tool Calling — Model determines how functions are executed
The LLM can choose from a set of tools (functions, APIs) and decides when and how to call them. This is where "agency" truly begins. Workflow: Input → LLM ← [Tool 1, Tool 2, Tool N] → Output. Example: Perplexity AI. You ask a question; the model decides whether to search the web, look at a specific URL, do math, or answer from memory. The tool selection decision belongs to the model.
L4a
Multi-Step Agent — Model controls iteration and program continuation
A Meta-Agent orchestrates multiple Sub-Agents, controlling whether to continue, repeat, or stop the overall process. The model decides when the task is "done enough." Workflow: Input → Meta-Agent → [Sub-Agent 1 ... Sub-Agent N] → Meta-Agent evaluates → Output or loop. Example: A research agent that spawns sub-agents for each section of a report, evaluates quality, and decides whether to regenerate sections.
L5
Fully Autonomous — Model creates and executes new code and functions
The most powerful and risky level. The LLM can write new code, create new tools, and execute them — extending its own capabilities at runtime. Workflow: Input → LLM Generator creates solution → LLM Validator runs it → Feedback loop → Output. Example: Devin (AI software engineer). Given a task, it writes code, executes it, sees errors, rewrites code, installs dependencies it didn't have before, and ships a working solution.

Module 03 · Architecture
9 Agentic
Workflow Patterns
The fundamental design patterns used in production agentic systems. Every real-world agent is built from one or more of these nine patterns. Understanding them is the difference between architecting agents and just prompting LLMs.
Pattern 1: Prompt Chaining

Concept: Decomposes a complex task into a linear sequence of smaller LLM calls where each call processes the output of the previous one. A Gate node validates each step — if validation fails, the chain exits with an error rather than producing bad output.

Why use it: Complex tasks done in one prompt produce lower quality results. Breaking into steps lets each LLM call specialize, resulting in higher overall quality.

Step 1: Extract key clauses. Gate: Verify all required clauses present. Step 2: Analyze each clause for risk. Gate: Check risk scoring completeness. Step 3: Generate summary report with recommendations.
Pattern 2: Parallelization

Concept: Run multiple LLM calls simultaneously — either splitting a task into independent sections (sectioning) or running the same task multiple times to aggregate/vote on results. An Aggregator node combines outputs.

Two sub-types: (1) Sectioning — divide a task into parallel workstreams. (2) Voting — run the same prompt N times and take the majority/best result.

Pattern 4: Evaluator-Optimizer

Concept: One LLM (Generator) creates output; a second LLM (Evaluator/Validator) scores it and provides feedback. If rejected, the generator tries again incorporating the feedback. This loop continues until quality passes a threshold.

Key insight: Critique is easier than generation. A small LLM can evaluate quality effectively, enabling a powerful quality-control loop without expensive retraining.

Generator writes unit tests for a function. Evaluator runs tests → checks coverage → scores. If coverage < 80% or any test fails: "Missing edge case for null input, retry." Generator revises. Loop until all tests pass with >90% coverage.

Patterns 5–9: Overview
Pattern 5
Routing
Input classified → directed to specialized handler. Router node decides which path. Best for: Customer support (billing vs. tech vs. returns), multi-domain Q&A systems. Example: Intercom routes your message to the right department using LLM classification.
Pattern 6
Autonomous Workflow
Agent operates in a feedback loop with its environment. LLM takes an action → observes result → decides next action → repeat until goal met or stopped. Best for: Computer-using agents (CUA), robotic process automation. Example: An agent that books flights by controlling a browser.
Pattern 7
Reflexion
Agent generates a response → writes a self-critique → revises → repeat N times. Learns through verbal reflection, not gradient descent. Best for: Complex data monitoring, full-stack app building. Limitation: can get stuck in self-criticism loops. Example: An agent writing a speech that iteratively critiques its own rhetoric.
Pattern 8
Rewoo
A Planner generates a full task list upfront with variable slots → a Solver fills in variables as it executes → Worker completes each task with filled-in context. Optimizes tool use. Best for: Deep research, multi-step Q&A. Reduces LLM calls by pre-planning the variable chain.
Pattern 9
Plan and Execute
Planner creates subtask list → individual Task Agents solve each one → results sent back to Planner which can replan failed tasks. Best for: Business process automation, data pipeline orchestration. Unlike Orchestrator-Worker, the Planner can update the plan based on intermediate results. Example: An agent automating a month-end close process.
Module 03 · Architecture
6 Agentic
Design Patterns
The six dominant patterns used by production AI agent products today — each representing a distinct architectural philosophy. Understanding these lets you reverse-engineer any agent system you encounter.
Pattern 1: ReAct Agent

Concept: Reasoning + Acting. The agent alternates between thinking (using the LLM to reason about the current state) and acting (calling a tool). After each action, it observes the result and reasons again. This creates a Think → Act → Observe → Think loop.

Concrete Example: Research Agent
User: "What are Tesla's Q3 2025 revenue and how does it compare to Q3 2024?"
Think: "I need Tesla's Q3 2025 revenue. I'll search for it." Act: Call search("Tesla Q3 2025 earnings"). Observe: "$25.2B revenue." Think: "Now I need Q3 2024 for comparison." Act: Call search("Tesla Q3 2024 earnings"). Observe: "$25.18B revenue." Think: "I have both numbers. I can now answer." Output: "Q3 2025 was $25.2B vs $25.18B in Q3 2024 — essentially flat YoY."
Used By
Most commercial AI agent products use ReAct as their foundational architecture. It's the default pattern in LangChain, LlamaIndex, and most agent frameworks because it's simple, debuggable, and works well with modern LLMs that support tool calling.
Pattern 2: CodeAct Agent

Concept: Instead of calling tools via JSON function calls, the agent writes and executes Python code directly. The code IS the action. This allows agents to perform arbitrary computations, not just call predefined functions. Developed and popularized by Manus AI.

Why it's powerful: JSON tool calling limits you to predefined function signatures. Python code has no such limits — the agent can do math, data manipulation, file operations, API calls, and dynamic logic all in one code block.

Patterns 3–6: Overview
Pattern 3 · Cursor
Modern Tool Use via MCP
Agent leverages the Model Context Protocol (MCP) to access tools — search (Kagi), cloud (AWS), databases — without hardcoded integrations. MCP acts as a universal adapter. The agent discovers available tools dynamically at runtime. Enables plug-and-play tool ecosystems. Used by Cursor IDE.
Pattern 4 · Open Serve AI
Self-Reflection (Reflexion)
Main LLM generates a first draft → a Critic LLM evaluates and writes critique → Generator revises → repeat. Loop continues until quality threshold met or max iterations. The agent literally writes "what was wrong with my last answer" and uses it to improve. Great for writing, coding, and analysis quality.
Pattern 5 · Gemini Deep Research
Multi-Agent Workflow
Multiple specialized sub-agents work in parallel on different aspects of a task. A coordinator orchestrates, collects results, and merges into a unified output via an Aggregator LLM. Greater precision than single-agent architectures. Sub-agents can use different models and different tools optimized for their specialty.

Agentic RAG

Module 05 · RAG & Tools
Tool Using
Strategies
How AI agents should strategically manage tool calls to reduce cost, latency, and complexity. Four tactical patterns for production-grade tool use.
4 Tool Strategies Compared
Strategy 1
Tool Caching
What: Cache the results of frequently-called tools so the agent doesn't need to call the API again for the same query. When to use: Agentic search systems that constantly call the same Search API. Example: A customer support agent that searches the same FAQ database hundreds of times a day — cache the top 100 FAQ embeddings in memory. Cost reduction: up to 90% fewer API calls for cached trajectories.
Strategy 2
Parallelization
What: Call multiple tools simultaneously instead of sequentially. When to use: Single Agent RAGs and Data Analyst agents needing real-time data from multiple sources. Example: A financial analysis agent that simultaneously calls stock price API + earnings DB + news API, instead of calling one at a time. Latency improvement: 3 sequential 500ms calls (1.5s) → 3 parallel calls (500ms).
Strategy 3
Tool Chaining
What: Run tools in a defined sequence where each tool's output feeds the next. When to use: Workflows where order matters — travel planning, document processing. Example (Travel Agent): (1) Search destination → (2) Get flight details → (3) Update flight + hotel doc. Sequential because you need destination info before searching flights, and flight info before booking hotels. Guarantees correct data dependencies.
Strategy 4
Grouping
What: Bundle commonly co-used tools into named groups. The agent calls the group, not individual tools, reducing prompt complexity and token usage. When to use: Multi-Agent RAGs or multi-tool products (Cursor). Example: Banking support agent groups: "Email Drafting + Slack Update + DB Retrieve" as Group 1. Calling Group 1 executes all three. Reduces tool description tokens, improves LLM tool selection accuracy.

Multi-step process where step B needs step A's outputTool ChainingEnforce correct data dependencies

Module 04 · Building
7-Step Roadmap
for Scalable AI Agents
A practical, battle-tested 7-step process for going from zero to a production-ready, scalable AI agent — from choosing the right LLM to coordinating multi-agent teams.
The Complete Roadmap
01
Pick an LLM
Decision criteria: Handles reasoning well, supports step-by-step logic (Chain of Thought), gives stable/consistent answers. Tip: Start with Llama 3, Claude Opus, or Mistral for open weights. Use Claude for tasks needing long-context reasoning. Use GPT-4o for instruction following. Use Llama for on-premise deployment. For niche multi-agents: You can even use smaller SLMs (Small Language Models) for specific sub-tasks — they're faster and cheaper.
02
Build Agent's Logic
Tell your agent "how to think": Should it reflect before answering? Should it plan steps or act immediately? Should it use tools when stuck? Tip: Start simple with ReAct (Reason + Act) or Plan-then-Execute. ReAct works for most cases. Plan-then-Execute is better when the task is highly structured and multi-step. Don't over-engineer — a ReAct agent with good tools beats a complex agent with bad tools every time.
03
Write Operating Instructions
Define the rules: How should it respond? When should it use external tools? What format should it reply in (JSON, markdown, prose)? Tip: Turn prompts into reusable instruction templates for better control and faster scaling. Store system prompts in a config file or DB — not hardcoded. Use versioned prompts so you can A/B test and rollback. Your system prompt IS your agent's brain stem — it should be treated like production code.
04
Add Memory
Why: AI forgets. Between sessions, nothing persists unless you build it. Three memory types: (1) Sliding window for short-term conversation context, (2) Summarization of older chats to compress context, (3) Long-term storage of key facts (user preferences, decisions) in a database. Tools: Mem0 or ZepAI for faster execution. PostgreSQL or Supabase as backing stores. For knowledge-heavy agents: add a knowledge graph layer.
05
Connect Tools & APIs
Let your agent "do" things: Search a database, fetch stock prices, call your CRM, send emails, book meetings. Tip: Just define what each tool does and when to call it — the LLM figures out how. Use MCP for plug-and-play tool integration. Start with 3–5 core tools. More tools = more context tokens = LLM struggles to choose. Add tools incrementally as you validate the agent works with each one.
06
Give it a Job
Good agents are specific: "Summarize user feedback and suggest improvements" is better than "Be helpful." Critical tip: Define what NOT to do more carefully than what to do. Limit the scope explicitly. An agent that refuses to do out-of-scope things is more reliable and safer than one that tries everything. Examples of scoping: "Only search our internal docs, not the web." "Never make purchases over $500 without approval." "Only respond in formal English."
07
Scale to Multi-Agent Teams
Don't stop at one: One agent gathers data → one interprets → one formats results. Tip: Use task-specific naming for agents for better optimization and debugging ("DataGathererAgent", "ReportFormatterAgent" — not "Agent1"). Choose communication protocol based on need: A2A (Google) for enterprise-grade, ACP (IBM/MCP-style) for rich multimodal, AGP (Cisco) for open market agent negotiation. Treat your agent team like a team of employees with specific roles.

Module 02 · Agent Concepts
RPA vs.
AI Agents
Robotic Process Automation (RPA) was the previous generation of automation. AI Agents are the next. Understanding the architectural differences explains why agents are so transformative — and when you still might prefer RPA.
Core Architectural Differences
RPA
Fixed Tool Selection
RPA uses predefined tool sequences hardcoded by a developer. "Always open Gmail, then Salesforce, then Sheets — in that order." If the process changes (Gmail UI update, new field in Salesforce), the bot breaks and needs manual reprogramming. No intelligence in tool selection — it's a script that mimics human clicks.
AI Agents
Auto Tool Selection
AI agents dynamically decide which tool to use based on the task and context. "This query needs a database lookup. This query needs a web search. This query needs both." The LLM evaluates the situation and picks the right tool. If a tool fails, the agent tries an alternative — no hardcoded fallbacks needed.
RPA
Rule-Based Execution
All logic is explicit: if [condition] then [action] else [other action]. Every possible scenario must be anticipated and programmed. Exception handling is manual. Works perfectly for perfectly predictable processes. Breaks immediately on anything unexpected.
AI Agents
Multi-Agent Workflow
A Control Agent orchestrates Retriever and Data sub-agents. Each sub-agent specializes. If the Control Agent encounters an unexpected situation, it reasons about it and decides a course of action — it doesn't crash. Handles ambiguity, exceptions, and novel situations gracefully.

When to Still Use RPA
RPA is still better for extremely high-volume, perfectly predictable, structured processes (e.g., extracting data from 10,000 identical invoices per day). The determinism and low per-transaction cost of RPA beats an LLM-based agent for pure, repetitive data entry work. Use agents when judgment, unstructured data, or variability is involved.

Architectures 4–7: Overview
Architecture 4
Human in the Loop
After the agent completes a task, execution pauses for Stack Approval — a human must approve before the agent proceeds. Uses an if/else gate: approved → continue; denied → respond with why. Best for: high-risk actions, financial transactions, legal document creation, any action that's hard to reverse.

Single Agent Calls Other Agents
A primary agent has a special tool: "Call another Agent." When a task is out of its specialty, it delegates to a specialist agent. The calling agent doesn't know the specialist's internals — it just sends a message and gets a result back. Enables specialization without full multi-agent complexity.
Architecture 6
Sequential Agents
Multiple agents connected in a chain. Agent 1 processes → output becomes input to Agent 2 → output becomes input to Agent 3. Like an assembly line. Each agent specializes in its step. Best for: document processing pipelines, content creation workflows (research → write → edit → format).
Architecture 7
Hierarchy + Parallel + Shared Tools
The most complex architecture. A Switch node routes to parallel agent trees. Multiple agent hierarchies share tools. A Merge node combines all parallel outputs. Best for: large-scale research, complex business process automation, competitive intelligence gathering where many tasks run simultaneously.
Module 03 · Architecture
Multi-Agent
Design Patterns
Six high-level design patterns for how multiple AI agents can be organized to solve problems together — from simple chains to autonomous self-organizing networks.
6 Multi-Agent Patterns
Pattern 1
Sequential
Agents in a pipeline — output of Agent A is input to Agent B. Linear and deterministic. Example: Customer complaint → Classifier Agent → Sentiment Agent → Response Agent → Email Agent. Best for: step-by-step processes where order matters and each step enriches the data for the next.
Pattern 2
Router
A Meta-Agent/Router receives the query and routes to specialized agents based on intent. Example: Travel Agent routes to Flight Agent or Hotel Agent. Customer Support routes to Billing Agent or Tech Agent. Key: the router's classification quality determines the entire system's quality.
Pattern 3
Parallel
A Divisor splits work across agents running concurrently. All agents run simultaneously, outputs merged. Example: Research query split into Web Search Agent + Database Search Agent, both run at once, merged into comprehensive answer. Massive latency improvement for independent subtasks.
Pattern 4
Generator
A Divisor creates multiple specialized generators for different aspects of the same output. Example: "Build a feature" → Coding Agent (code) + Documentation Agent (docs) + Debugging Agent (tests). All run, a Generator Agent combines into a complete, integrated output. Better than one agent doing everything.
Pattern 5
Network
Agents communicate with each other in a mesh — output of one can feed any other, not just the next in line. A Meta-Agent coordinates. Example: Coding Agent ↔ Debugging Agent can go back and forth multiple times until code passes all tests. More flexible than Sequential but harder to reason about.
Pattern 6
Autonomous Agents
Agents operate independently, communicating only when needed. No central coordinator. Example: Agent 1 monitors market data 24/7. Agent 2 generates trade signals when triggered. Each has its own goal, memory, and tools. This is the closest to true multi-agent autonomy. Hardest to control and debug.

Module 04 · Building
Anthropic's Guide:
Building Effective AI Agents
Anthropic's official framework for agent design — when to use agents vs. workflows, the 5 core patterns, 4 framework selection tips, and the critical guidance on when NOT to build agents.
Agents vs. Agentic Workflows (Anthropic's Definition)
Agentic Workflow
Fixed, Programmed Sequences
Characteristics: Rigid, suited for predictable tasks. Structured tasks like basic linear retrieval. The developer defines every step. When to use: When you know exactly what steps are needed, the task is repetitive, and you want reliability over flexibility. Example: A nightly report that always: fetches data → summarizes → emails stakeholders.
AI Agents
Autonomous Adaptation
Characteristics: Operates autonomously, adapts to task needs. Flexible, ideal for open-ended tasks. Suited for complex tasks like coding or real-time retrieval. When to use: When tasks vary significantly, require judgment, or involve ambiguous requirements. Example: A coding agent that reads a ticket and decides whether to refactor, add tests, or document first.

4 Tips Before Using Any Framework
T1
Ensure the framework aligns with your specific needs
Don't use LangChain because it's popular. Use it because its architecture fits your use case. LangGraph for stateful multi-agent systems. CrewAI for role-based teams. Agent SDK for Claude-heavy production deployments. Mismatched framework = fighting the tool instead of building the product.
T2
Evaluate performance in terms of accuracy, speed, and ability to handle complex tasks
Benchmark before committing. Run your most complex test cases through the framework. Measure: latency (is tool calling fast enough?), accuracy (does the agent make the right decisions?), reliability (does it fail gracefully?). A framework that works on toy examples may collapse on production-scale complexity.
T3
Prioritize frameworks with robust safety features
To ensure responsible agent behavior. Look for: built-in rate limiting, tool call approval mechanisms, output filtering, audit logging, and rollback capabilities. Agents that can take irreversible actions (send email, make payments, delete data) need safety guardrails built into the framework — not bolted on later.
T4
Confirm the framework integrates well with existing systems
E.g., APIs, databases. The best agent framework is one your team can actually connect to your existing stack. Check: does it support your DB? Does it work with your auth system? Can you add custom tool connectors? A framework with 100 pre-built integrations beats a "better" one that requires rewriting your entire infrastructure.
When Should You Build AI Agents? (Anthropic's Guidance)
Start Simple First
Start with the simplest solution — a single LLM call or basic workflow — and only move to agents when workflows fall short. Many problems that seem to need agents can be solved with well-designed prompts and simple pipelines. Every added layer of agent complexity is also added cost, latency, and debugging surface.
Agents Need Extensive Testing
Agents need extensive testing due to their autonomy. Make sure you have the right resources for testing. An agent that works 95% of the time is not production-ready for high-stakes tasks. You need: unit tests per tool, integration tests for tool chains, red-teaming for adversarial inputs, and monitoring in production.
Workflows First, Agents Second
Workflows are often implemented with a few lines of code, so if your workflow is not complex, refrain from using complex frameworks to build agents. The overhead of an agent framework (additional prompts, tool routing, memory management) adds cost and latency. Only add this overhead when you genuinely need the flexibility.
Module 04 · Building
AI Agent
Learning Roadmap
The complete learning path from "I know some Python" to "I can build production multi-agent systems" — organized in a logical dependency graph with specific topics at each node.
Level 1: Basics of ML and GenAI
Node 1 — Start
Basics of Python / TypeScript
Topics: Data types, control structures (if/for/while), File I/O, Network requests (requests/fetch). Why first: Every AI framework is Python or TS. You can't build agents without fluent Python. Target: write a working REST API client in Python from scratch.
Node 2
Basics of Machine Learning
Topics: Types of ML (supervised, unsupervised, RL), Neural Network architecture, Reinforcement Learning basics. Why: Agents are built on LLMs which are built on these concepts. Understanding training/inference distinction is critical for knowing why agents behave as they do.
Node 3
Basics of LLMs
Topics: Transformers (attention mechanism), Mixture of Experts (MoE), Fine-tuning approaches, Context Window. Why: The LLM is the brain of your agent. Understanding context windows explains why memory matters. Understanding transformers explains why prompt structure affects output quality.
Node 4
Basics of Prompt Engineering
Topics: Chain of Thoughts, Graph of Thoughts, Few Shot / Zero Shot, Role-based prompts. Why: Your system prompt IS your agent's instructions. Bad prompting = bad agents. Chain of Thought alone can increase accuracy by 40%+ on complex reasoning tasks.
Node 5
Basics of API Wrappers
Topics: API Types (REST, GraphQL, gRPC), GPT-Wrappers (Anthropic SDK, OpenAI SDK), File I/O, Authentication (API keys, OAuth). Why: Tools in your agent are API calls. You need to fluently connect to any API — this is the plumbing of agent tool use.
Level 1 (continued): Deep Dive into RAGs and AI Agents
Node 6
Basics of RAGs
Topics: Embeddings (text → vectors), Vector Store (Chroma, Qdrant), Retrieval Models, Generation Models. Why: RAG is the core technique for grounding agents in real data. Without it, your agent can only use what's in its training data or context window. With it, agents can access any document corpus.
Node 7
Basics of AI Agents
Topics: Types of Agents (reactive, goal-based, learning), Design Patterns (ReAct, CodeAct), Tools and MCP, Agent Memory. Why: Before building, you need a solid mental model. This node gives you the conceptual foundation for all the architectural decisions ahead.
Node 8
AI Agent Frameworks
Topics: Orchestration (LangGraph, CrewAI), Planning (ReAct, Plan-Execute), Feedback Loop implementation, Streaming responses. Why: Frameworks abstract the complex plumbing. Instead of writing tool routing from scratch, you define agents in a framework and it handles execution, retry, logging, and streaming.
Node 9
Multi-Agent Systems
Topics: Types of MAS (Sequential, Parallel, Network), Communication Patterns, Hand-offs between agents, A2A Protocol. Why: Single agents hit limits. Real enterprise automation requires coordinated teams of agents. This is where the most powerful (and complex) systems live.
Node 10 — End
Evaluation and Observability
Topics: Metrics (accuracy, RAGAS scores, latency), Logging (trace every LLM call), Latency profiling, Stress testing. Why: You can't improve what you don't measure. Production agents need: dashboards showing success rates, logs for debugging failures, latency tracking per tool, and load testing before scaling. This is what separates hobby projects from production systems.
